# BankongSeton v1.0 — Milestone Audit

**Date:** 2026-03-02 (updated from 2026-03-01)
**Scope:** 11 phases, 41 registered v1 requirements + 4 unregistered WEB requirements
**Overall Status: `gaps_found`**

---

## Executive Summary

All 11 phases completed with passing or signed-off phase verifications. However, two integration audit passes reveal **7 functional failures** preventing v1.0 from being production-ready.

**From the first audit pass (2026-03-01) — cross-phase wiring failures:**
1. **APP-03 / TD-03** — Every cashier POS receipt shows ₱0.00 for balance history (`BalanceBefore` column missing from cashier writes, causing schema column shift).
2. **NOTF-01 / TD-02** — FCM push notifications never fire from the cashier POS payment path; also blocked on fresh deploys by `migrate_users_schema()` not being called at startup.

**From the second audit pass (2026-03-02) — NFC + security gaps across all 11 phases:**
3. **NFC-03** — NFC tap-to-pay broken end-to-end: `NfcRegistrationResponse` field names don't match backend response → `currentToken` is always null → every NFC tap returns SW_NO_TOKEN (0x6A82).
4. **NFC-04** — NFC device token has no path from Android HCE to cashier HTTP header (`X-Device-Token` required by `/api/nfc/pay` but APDU only transmits the card token).
5. **SEC-01** — Cashier credentials `cashier`/`cashier123` hardcoded in source code (cannot be rotated without code change and redeploy).
6. **SEC-02** — `cashier_routes.py` JWT_SECRET has publicly-known hardcoded fallback `"bangko-jwt-secret-2026"` (any attacker can forge cashier JWTs if env var is absent).
7. **WEB-02** — NFC simulation UI not implemented (zero code in any dashboard template despite being a Phase 07.1 deliverable).

**Note on BUG-01:** The 2026-03-01 audit identified `cashier_request_card` WebSocket event emitted with no handler. The second audit pass (covering the full 11-phase codebase with the Arduino hardware path and `web_app.py` as primary cashier) reclassified: the Arduino bridge cashier path is hardware-dependent and the `web_app.py` `POST /cashier/process` path is the actual deployed payment path. BUG-01 remains as a tech debt item (TD-01) but is not a FAIL-gate blocker for the web-deployed cashier.

**Recommendation: Do not declare v1.0 complete. Run `/gsd:plan-milestone-gaps` to plan fixes for the 7 blockers.**

## Cross-Phase Wiring Assessment

### Working correctly
- `normalize_card_uid()` (Phase 2) used by Phase 1, 3, 5 — all 3 backend entry points
- `get_logger()` (Phase 2) used across all backend modules in Phases 1–5
- `CardReaderState` singleton (Phase 2) used by admin_dashboard.py (Phase 1)
- All Phase 4 student app API calls wired to correct endpoints
- Phase 3 admin product routes all use `ensure_products_sheet()` correctly
- Phase 07.1 web dashboard: 8 socket.io event handlers all wired to Flask-SocketIO emitters
- Phase 4 FCM: student_app_v2 FCMService → Firebase → api_server.py fcm_sender.py all wired
- Phase 3 product management → cashier POS display confirmed fully wired

### Missing connections (causing the 7 blockers)
- `NfcRegistrationResponse.virtual_token` → backend sends `virtual_card_token` → token always null
- APDU protocol (HCE) transmits only card token → `/api/nfc/pay` requires additional `X-Device-Token` header
- `cashier_routes.py JWT_SECRET` has hardcoded public fallback (not using env var enforce pattern)
- Cashier credentials hardcoded in source (not env-var-backed)
- `send_low_balance_push()` not called from `cashier_routes.py complete_sale()`
- `migrate_users_schema()` not called at api_server.py startup
- `BalanceBefore` column not written by cashier_routes.py (7-col vs 8-col schema)
- WEB-02 NFC simulation: specified, verified in 07.1, but never implemented

### Additional handoff misalignments (partial failures)
- Timestamp format: backend writes `%Y-%m-%d %H:%M:%S`; Android reads `yyyy-MM-dd'T'HH:mm:ss`
- Transaction type: NFC backend writes `"NFC Purchase"`; Android adapter only matches `"Purchase"`
- `ensure_products_sheet()`: admin routes use it; cashier route uses `db.worksheet()` directly
