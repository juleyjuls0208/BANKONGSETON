# Phase 32.1 — NFC Fix (INSERTED URGENT)
# Context for researcher + planner

## What this phase is about

Two problems that must be fixed before the canteen can accept physical student
payment cards:

1. **NTAG215 physical cards are not detected** by the PN532 Arduino reader.
   The PN532 RF layer returns NbTg=0 (no target) when an NTAG215 card is tapped,
   even though the same card reads on iPhone NFC Tools and generic RFID testers.
   HCE phone payments (Android) continue to work.

2. **ArduinoBridge does not support two simultaneous readers.** Both the R3 and
   R4 readers are deployed at the canteen. The bridge currently accepts a single
   serial port and uses an old `<CARD|...>` wire format that doesn't match what
   the sketches emit. Two-reader support is in scope for this phase.

## Diagnosed root cause — NTAG215 not detected

**Diagnostic evidence** (pn532_packetbuffer printed on failed NTAG215 tap):

```
SCAN: no card — raw buf: 00 01 00 04 08 04 03 91 68 31 00 00 00 00 00 00
```

- `pn532_packetbuffer[0] = 0x00` → `NbTg=0` → PN532 RF layer detected nothing
- `pn532_packetbuffer[2] = 0x00` → confirmed: PN532 truly found no target
- Bytes `[1..9]` are stale from a previous successful HCE phone scan

The PN532 is responding over SPI (firmware version found, SAMConfig succeeds),
but it is not generating the RF field correctly for NTAG215 detection.

**Most likely cause:** `nfc.setRFField(1, 1)` in the R4 scan loop uses
`autoRFCA=1` (RF collision avoidance). This flag is for NFC active/peer-to-peer
modes, not for passive ISO 14443-3A reading. With `autoRFCA=1`, the PN532 may
suppress its own RF output during the sensing phase, preventing NTAG215 from
powering up. The correct call for passive card reading is `setRFField(0, 1)` —
RF on, no autoRFCA.

**Secondary factor:** `setPassiveActivationRetries(0x0A)` = 10 retries may be
too few for NTAG215's double-cascade anticollision (7-byte UID). Mifare Classic
4-byte single-cascade may succeed in 10 retries but NTAG215 may need more.

**What works vs. what fails:**

| Target              | Works?   | Notes                              |
|---------------------|----------|------------------------------------|
| Android HCE phone   | Yes      | 4-byte UID, APDU path              |
| NTAG215 physical    | No       | 7-byte UID, NbTg=0                 |
| Mifare Classic card | Untested | Likely broken too on R4            |
| iPhone NFC Tools    | Yes      | Card is electrically valid         |
| Generic RFID tester | Yes      | Card is electrically valid         |

## Diagnosed root cause — ArduinoBridge format mismatch

The sketches emit:
```
NFC|<48-char-token>
CARD|<uid-hex>
```

The current `arduino_bridge.py` reads the old format:
```
<CARD|ABCD1234>   ← angle-bracket delimited, uppercase, no NFC| support
```

The bridge also only holds a single `self.arduino` serial instance and cannot
receive input from two readers simultaneously.

## Fix scope

### Arduino sketches (both R3 and R4)

**Full rewrite of the scan loop in both sketches.**

Required changes:
1. Fix `setRFField` parameters: `setRFField(1, 1)` → `setRFField(0, 1)`
   (RF on, no autoRFCA — correct for passive ISO14443A)
2. Tune or remove `setPassiveActivationRetries` — test higher values (≥0x50)
   or remove and use PN532 default
3. Preserve UID-length dispatch:
   - `uidLen == 4` → APDU path (HCE phone)
   - `uidLen != 4` → direct UID path (NTAG215, Mifare physical cards)
   This logic is already correct in the R4 sketch; backport it to R3
4. Preserve RF field toggle between scans (off→on for state reset),
   but fix the `setRFField` parameters used in the toggle
5. Retain all delivery logic unchanged:
   - R4: WiFi POST with serial fallback (`deliver()`)
   - R3: serial only (`CARD|<uid>` and `NFC|<token>`)
6. Retain SCAN diagnostic output (raw `pn532_packetbuffer` dump on no-detection)

### ArduinoBridge (`backend/dashboard/arduino_bridge.py`)

**Multi-reader support + wire format alignment.**

Required changes:
1. Support multiple serial ports simultaneously (one per connected Arduino)
2. Parse both wire formats:
   - `NFC|<token>` → route as HCE token payment
   - `CARD|<uid>` → route as UID-based card payment
3. Remove the old `<CARD|...>` angle-bracket format (no longer used)
4. Each reader runs its own read thread; payments from either reader are
   processed identically by the backend
5. Connection setup in Flask (`__init__.py` or wherever serial is initialized)
   must enumerate and connect all available Arduino serial ports

## Key constraints

- R3 has no WiFi — serial is the ONLY delivery path; do not add WiFiS3.h
- R4 already has WiFi POST + serial fallback; keep both paths
- The 48-char HCE token format is unchanged
- `SCAN_COOLDOWN_MS = 1500` is intentional; do not remove
- Both sketches share the same PN532 wiring (SPI, D10/D11/D12/D13)
- Baud rate is 9600 on both — ArduinoBridge expects this rate
