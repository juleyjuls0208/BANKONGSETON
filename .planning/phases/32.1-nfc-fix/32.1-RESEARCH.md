## Summary

Phase 32.1 fixes two hardware/firmware blockers preventing physical NTAG215 student cards from
working at the canteen. The root causes are fully diagnosed and confirmed by direct code inspection.
No new libraries are needed — all changes are surgical edits to existing files.

**Problem 1 (Arduino firmware):** The PN532 RF field toggle in both sketches uses
`setRFField(1, 1)` (autoRFCA=1) which is for NFC peer-to-peer active mode, not for passive
ISO14443A card reading. With autoRFCA=1, the PN532 suppresses its own RF field during the
sensing phase, so NTAG215 never powers up → NbTg=0. The fix is `setRFField(0, 1)` (RF on,
no autoRFCA). The R4 sketch has this bug in **two places**; the R3 sketch is missing the RF
toggle entirely and needs it added. R3 also needs UID-length dispatch backported from R4 so
NTAG215 cards skip the APDU path.

**Problem 2 (ArduinoBridge):** The current `arduino_bridge.py` parses the obsolete
`<CARD|ABCD1234>` angle-bracket format. The sketches actually emit `CARD|<uid>` and
`NFC|<token>`. The bridge also holds only one `serial.Serial` instance; the canteen has both
R3 and R4 deployed. The fix is to add a `_parse_line()` method handling both current formats,
a `_start_serial_thread()` that loops on `readline()`, and a `start_background_listener()` for
continuous operation. The existing `read_card_with_timeout()` API must be preserved for
`dashboard_core.py` compatibility.

**Primary recommendation:** Make all changes in one PR: fix both sketches and rewrite
ArduinoBridge. The sketch fixes are the highest-priority because they unblock physical card
payments; the bridge fix is required to receive output from both readers.

## Architecture Patterns

### PN532 `setRFField` API (Elechouse library)

**Signature:** `nfc.setRFField(uint8_t autoRFCA, uint8_t RFonOff)`

| Parameter | 0 | 1 |
|-----------|---|---|
| `autoRFCA` | RF collision avoidance OFF (passive card mode) | RF collision avoidance ON (active/peer-to-peer) |
| `RFonOff` | RF field OFF | RF field ON |

**Correct calls for passive ISO14443A (cards):**
```cpp
nfc.setRFField(0, 0);  // RF off  — field off to reset NTAG state
delay(50);
nfc.setRFField(0, 1);  // RF on   — no autoRFCA, passive mode
delay(10);
```

**Bug in current R4 (lines 443, 555):**
```cpp
nfc.setRFField(1, 1);  // WRONG: autoRFCA=1 suppresses RF during sensing
```

### R4 Sketch — Exact Lines to Change

| Location | Line(s) | Current | Fix |
|----------|---------|---------|-----|
| Scan loop RF toggle | 443 | `nfc.setRFField(1, 1);  // RF on` | `nfc.setRFField(0, 1);  // RF on, no autoRFCA` |
| APDU-failed reset | 555 | `nfc.setRFField(1, 1);  // RF on` | `nfc.setRFField(0, 1);  // RF on, no autoRFCA` |

The `setRFField(0, 0)` calls on lines 441 and 553 are **already correct** — do not change them.

### setPassiveActivationRetries Tuning

The CONTEXT identifies this as a secondary factor. NTAG215 uses double-cascade anticollision
(7-byte UID: CT+3bytes+CT+3bytes+1byte) vs Mifare Classic single-cascade. The PN532 default
(no explicit call) uses 0xFF (infinite retries, controlled by timeout). Options:

| Approach | Value | Trade-off |
|----------|-------|-----------|
| Remove call entirely | PN532 default (0xFF) | Reliable detection; loop bounded by `NFC_TIMEOUT_MS` timeout |
| Increase to 0x50 | 80 retries | More reliable than 0x0A; still terminates quickly on no-card |
| Current R4 value | 0x0A (10 retries) | Too few for NTAG215 double-cascade |

**Recommendation:** Remove `setPassiveActivationRetries` call entirely from both sketches.
The `NFC_TIMEOUT_MS` parameter to `readPassiveTargetID` bounds the blocking time. The comment
in the R4 sketch (lines 402–408) explaining the 0x0A choice applies to HCE phones, not NTAG215.
With `setRFField(0,1)` fixed, the retry count matters less, but removing it is the safest
approach to guarantee NTAG215 detection.

> **CONTEXT says:** "Tune higher values (≥0x50) or remove and use PN532 default" — both are valid. Removing is simpler and lower risk.

**Required ArduinoBridge interface (backward-compatible):**
```python
class ArduinoBridge:
    def __init__(self, arduino_serial, socketio):
        # single-port ctor (unchanged for dashboard_core.py compatibility)
        self.arduino = arduino_serial
        self.socketio = socketio
        self.reading_active = False
        self.current_callback = None
        self.timeout_seconds = 5
        self._start_serial_thread()  # called in ctor; patched in tests

    def _start_serial_thread(self):
        """Start background thread reading self.arduino continuously."""
        thread = threading.Thread(target=self._serial_loop, daemon=True)
        thread.start()

    def start_background_listener(self):
        """dashboard_core.py calls this after ctor. May be no-op if _start_serial_thread
        already started in ctor, or can start/restart the thread."""
        pass  # or start if not already running

    def _serial_loop(self):
        """Loop: read lines from serial port, call _parse_line."""
        while self.arduino and self.arduino.is_open:
            try:
                if self.arduino.in_waiting > 0:
                    line = self.arduino.readline().decode('utf-8', errors='ignore').strip()
                    if line:
                        self._parse_line(line)
                time.sleep(0.05)
            except Exception as e:
                break  # exit loop if serial error

    def _parse_line(self, line: str):
        """Parse one serial line and emit appropriate SocketIO event."""
        if line.startswith("NFC|"):
            token = line[4:]
            self.socketio.emit("nfc_payment", {"token": token})
            self._post_nfc_payment(token)
        elif line.startswith("CARD|"):
            uid = line[5:]
            self.socketio.emit("card_read", {"success": True, "uid": uid})
            if self.reading_active and self.current_callback:
                self.reading_active = False
                self.current_callback(uid)
        elif line.startswith("ERROR|"):
            error = line[6:]
            self.socketio.emit("nfc_payment_result", {"success": False, "error": error})

    def _post_nfc_payment(self, token: str):
        """POST token to /api/nfc/pay. Requires CASHIER_JWT env var."""
        jwt = os.environ.get("CASHIER_JWT", "")
        base_url = os.environ.get("API_BASE_URL", "http://127.0.0.1:5001")
        if not jwt:
            # log warning, do not raise
            return
        requests.post(
            f"{base_url}/api/nfc/pay",
            json={"token": token},
            headers={"Authorization": f"Bearer {jwt}"},
            timeout=5,
        )

    def read_card_with_timeout(self, callback, timeout=5):
        """Existing API — used by dashboard_core.py SocketIO handler."""
        if not self.arduino or not self.arduino.is_open:
            self.socketio.emit('card_error', {'message': 'Arduino not connected'})
            return False
        self.reading_active = True
        self.current_callback = callback
        self.timeout_seconds = timeout
        # Background loop already running; it will call callback via _parse_line
        # Start a timeout thread to emit card_timeout if no card arrives
        t = threading.Thread(target=self._timeout_watcher, daemon=True)
        t.start()
        return True

    def _timeout_watcher(self):
        import time
        time.sleep(self.timeout_seconds)
        if self.reading_active:
            self.reading_active = False
            self.socketio.emit('card_timeout', {
                'message': f'No card detected within {self.timeout_seconds} seconds'
            })

    def cancel_reading(self):
        self.reading_active = False
        self.current_callback = None
```

### Multi-Port Support — Design Decision

**CONTEXT says:** "Each reader runs its own read thread; payments from either reader are
processed identically by the backend."

**Two approaches:**

| Approach | Description | Complexity |
|----------|-------------|------------|
| A — Two ArduinoBridge instances | Create one `ArduinoBridge` per port; each has its own thread | Low — existing class handles it; `dashboard_core.py` already creates bridge per port |
| B — One ArduinoBridge, multiple ports | Add `ports` list to constructor; spawn one thread per port | Medium — constructor API change |

**Recommendation: Approach A** (two instances). `dashboard_core.py` already creates a bridge
when `/api/serial/connect` is called. Admin connects each reader separately via the UI. Since
both readers are already connected at the canteen, admin would hit connect for port 1 and then
port 2 — each creates its own `ArduinoBridge` and background thread. The `app` object could
store `app.arduino_bridges = []` if needed, or the existing single `app.arduino_bridge` slot
can be updated to hold a list.

**CONTEXT says:** "Connection setup in Flask (`__init__.py` or wherever serial is initialized)
must enumerate and connect all available Arduino serial ports." This implies auto-enumeration
at startup, not manual connect. However, the current architecture uses a UI-triggered
`/api/serial/connect` endpoint. For minimal risk: keep UI-triggered connect, just fix the
bridge class itself. Auto-enumeration is a "nice to have" that the planner can decide.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| UID length → tag type detection | Custom card-type registry | PN532's returned `uidLen` (4 = HCE, 7 = NTAG/Mifare physical) | ISO14443-3A cascading levels directly encode UID length |
| Serial line buffering | Custom ring buffer | `pyserial.readline()` | Already handles partial lines and CR/LF |
| Thread safety for `reading_active` | Custom locking | Simple `threading.Lock` on `reading_active` flag | Flag is only toggled, not read-modify-write |
| PN532 RF timing | Custom delays | PN532 library's built-in timing + 50ms off / 10ms on per existing R4 pattern | PN532 firmware handles anticollision timing |

## Common Pitfalls

### Pitfall 1: Confusing `setRFField(autoRFCA, RFonOff)` argument order
**What goes wrong:** Swapping arguments → wrong behavior (RF stays off, or collision avoidance
turned on when it should be off).
**Why it happens:** The PN532 datasheet names these differently from the Elechouse wrapper.
**How to avoid:** Always write as `setRFField(0, 1)` for "RF on, passive mode" and confirm
from the function signature in `PN532.h`.
**Warning signs:** NbTg=0 persisting after fix; scan works sometimes but not reliably.

### Pitfall 2: Changing `setRFField(0, 0)` calls (they're correct)
**What goes wrong:** Changing the "RF off" calls while fixing the "RF on" calls.
**How to avoid:** Only change the second argument to `setRFField` in the `// RF on` lines.
R4 has 4 `setRFField` calls total: 2 off (correct), 2 on (wrong → fix).

### Pitfall 3: R3 `NFC_TIMEOUT_MS` is 500ms vs R4's 5000ms
**What goes wrong:** Adding RF toggle + UID dispatch but forgetting that R3's `NFC_TIMEOUT_MS`
is 500ms — adequate for phones but may be marginal for NTAG215. R3 loops rapidly (no WiFi
overhead), so the low timeout is acceptable for quick re-polling.
**How to avoid:** Keep 500ms for R3. The fix is the RF toggle and UID dispatch, not timeout.

### Pitfall 4: `_start_serial_thread` must be patchable (tests depend on it)
**What goes wrong:** Inlining the thread start directly in `__init__` without the separate
`_start_serial_thread` method breaks the test fixture which patches `_start_serial_thread`.
**How to avoid:** Always call `self._start_serial_thread()` from `__init__`; keep as a
separate named method so `patch.object` can intercept it.

### Pitfall 5: `_parse_line` for `CARD|` must also unblock `read_card_with_timeout`
**What goes wrong:** Background thread emits `card_read` but the `reading_active` callback
path is never triggered → cashier POS hangs waiting for card.
**How to avoid:** In `_parse_line`, when `CARD|` is received and `self.reading_active` is
True, call `self.current_callback(uid)` and clear `reading_active`.

### Pitfall 6: UID validation in `dashboard_core.py` only accepts 8 or 14 hex chars
**What goes wrong:** A 7-byte NTAG215 UID becomes a 14-character hex string. `dashboard_core.py`
`UID_PATTERN` is `^[0-9A-Fa-f]{8}$|^[0-9A-Fa-f]{14}$` (already accepts 14 chars). But
`cashier_routes.py` has `UID_PATTERN = re.compile(r'^[0-9A-Fa-f]{8}$')` — only accepts 8!
**Impact:** NTAG215 UIDs (14 hex chars) will be rejected by the cashier payment route.
**How to avoid:** Update `cashier_routes.py` UID_PATTERN to also accept 14-char UIDs.
This is a required fix for end-to-end NTAG215 payments to work.
**Warning signs:** NTAG215 reads correctly at serial level but payment fails with 400 error.

### Pitfall 7: `admin_dashboard.py` still uses `<CARD|...>` angle-bracket format
**What goes wrong:** `admin_dashboard.py` line 1194 still parses `<CARD|ABCD1234>` format.
This is the legacy `admin_dashboard.py` code path (not `dashboard_core.py`). Payments through
the admin dashboard card registration path will silently fail to parse NTAG215 UIDs.
**How to avoid:** If the admin dashboard card registration uses this path, update it too.
However, the CONTEXT says to focus on `arduino_bridge.py`. Flag for planner to confirm scope.

## Open Questions

1. **`setPassiveActivationRetries` — remove vs 0x50?**
   - What we know: CONTEXT says "≥0x50 or remove." Removing is safest (uses PN532 default 0xFF; bounded by NFC_TIMEOUT_MS).
   - What's unclear: Whether 0x0A is truly causing secondary failures after the RF fix, or if the RF fix alone is sufficient.
   - Recommendation: Remove the call from both sketches. The NFC_TIMEOUT_MS timeout provides the scan duration bound.

2. **`cashier_routes.py` UID_PATTERN — in scope?**
   - What we know: `cashier_routes.py` line 26 only accepts 8-char UIDs. NTAG215 produces 14-char UIDs. This will silently reject NTAG215 payments at the cashier route.
   - What's unclear: CONTEXT says "not changing payment processing rules." But this is a routing bug, not a business logic change.
   - Recommendation: **Include this fix.** Without it, NTAG215 UIDs read correctly at serial level but are rejected at the cashier payment endpoint. The canteen cannot accept physical cards end-to-end.

3. **`admin_dashboard.py` legacy `<CARD|...>` parser — update or leave?**
   - What we know: `admin_dashboard.py` line 1194 parses old format. This is used for admin card registration, not cashier payments.
   - What's unclear: Whether admin card registration (register student ID / money card) happens through this old code path or through `dashboard_core.py`.
   - Recommendation: Flag for the planner. If admin card registration is still used (R4/R3 both emit `CARD|` now), this path silently fails. Low risk to leave for now since `dashboard_core.py` at line 412 already handles `CARD|` format.

4. **Auto-enumeration of serial ports at startup?**
   - CONTEXT says: "Connection setup in Flask must enumerate and connect all available Arduino serial ports."
   - Current architecture: Admin manually connects via `/api/serial/connect` UI.
   - Recommendation: The planner should decide. Auto-enumeration at app startup is desirable but carries risk (wrong port grabbed). Keep UI-triggered for this phase; note as future work.

5. **How many `app.arduino_bridge` slots?**
   - If two bridges are created (one per reader), `app.arduino_bridge = bridge` on second connect will overwrite the first.
   - Recommendation: Change to `app.arduino_bridges = []` list, or use dict keyed by port. The planner should decide the data structure.
