#### Endpoint entries to write

Use the style: `### METHOD /path`, then `**Auth:** ...`, then request/response sections.

All request/response shapes are verified in `10-RESEARCH.md`. Use them verbatim.

**10 routes:**
1. `GET /cashier/login` — No auth. Renders login page. Response: HTML (200).
2. `POST /cashier/api/login` — No auth. JSON body `{username, password}`. Response 200: `{"success": true}` + sets `jwt_token` HttpOnly cookie (8h). Response 401: `{"error": "Invalid credentials"}`.
3. `GET /cashier/` — JWT cookie required. Renders POS HTML page. Redirects to `/cashier/login` if not authenticated.
4. `GET /cashier/api/ports` — JWT cookie required. Response 200: `{"ports": ["COM3", "COM4"]}`. Returns `{"ports": []}` in web/cloud mode.
5. `POST /cashier/api/connect-arduino` — JWT cookie required. Body: `{"port": "COM3"}`. Response 200: `{"success": true, "message": "Connected to COM3"}`. Errors: 400 (port missing), 500 (connect failed), 503 (bridge unavailable).
6. `GET /cashier/api/products` — JWT cookie required. Response 200: `{"products": [{id, name, category, price, active}]}`. Returns all products including inactive ones; POS frontend filters. Response 503: service unavailable.
7. `POST /cashier/api/logout` — No auth (always succeeds). Response 200: `{"success": true}` + deletes `jwt_token` cookie.
8. `GET /cashier/api/lookup-student` — JWT cookie required. Query param `?q=<search>` (min 2 chars). Response 200: `{"students": [{id, name, balance, card_uid}]}`. Up to 10 matches. Errors: 503/500.
9. `POST /cashier/api/process-sale` — JWT cookie required. Body: `{"items": [...], "total": 20.0}`. Response 200: `{"status": "waiting_for_card", "message": "Please tap student card"}`. Side effect: emits WebSocket event `cashier_request_card`. Error 400: invalid sale data.
10. `POST /cashier/api/complete-sale` — JWT cookie required. Body (RFID): `{"card_uid": "ABCD1234"}` or (manual): `{"manual_student_id": "202501"}`. Response 200: `{"success": true, "new_balance": 200.0, "timestamp": "..."}`. Errors: see research (400/403/404/500/503 variants documented).

#### Change C — Add accurate FCM operational note

After the transaction table (and its closing `---` separator), add a new subsection:

```markdown
### Push Notifications

After a sale is completed, the cashier POS **does** send FCM push notifications to the student's registered Android device:

- **Purchase notification** — always sent after a successful `complete-sale` call.
- **Low-balance notification** — additionally sent if the student's new balance is below the configured `LOW_BALANCE_THRESHOLD` (default: ₱50).

Both notifications are **fire-and-forget**: a failure to deliver the push notification never blocks the transaction or triggers a rollback. If the student has not registered an FCM token (i.e., has never logged into the student app), no notification is sent — this is expected and silent.

> **Note:** FCM requires the student's `FCMToken` to be populated in the Users sheet. Tokens are registered automatically when a student logs into the Android app for the first time.
```
