# Phase 07.1: Web-Deployable Dashboard - Context

**Gathered:** 2026-03-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Create a web-deployable version of the admin dashboard (`web_app.py`) that runs on PythonAnywhere without Arduino/serial hardware. Admin and finance users can access all non-hardware features from any browser. Hardware-dependent sections (Arduino panel, cashier RFID tap) attempt hardware first and gracefully fall back to web-friendly alternatives. Deploying the mobile API server (`api_server.py`) is out of scope for this phase.

</domain>

<decisions>
## Implementation Decisions

### Cashier POS in web mode
- Cashier page auto-detects serial ports on load; if Arduino is available, hardware mode is used normally
- If no serial ports are detected, cashier automatically switches to **manual student ID / name lookup** mode
- In manual mode, cashier types a student ID or name to search and look up their account, then processes the payment
- Manual-entry transactions are tagged `'Manual'` in the transaction type column (distinct from `'Purchase'` via RFID)
- No separate cashier UI needed — same cashier page, different input method based on detection result

### Deployment target
- **PythonAnywhere** is the confirmed deployment host
- `wsgi.py` imports `from web_app import app as application` — `web_app.py` is the file to create
- `web_app.py` is a hardware-free version of `admin_dashboard.py` (strip Arduino-required startup code; keep conditional import pattern)
- `wsgi.py` path updated from `/home/bankoseton/FinanceDashboard` → `/home/bankoseton/BANKONGSETON`
- Flask serves static files directly (no Nginx/Apache proxy config needed on PythonAnywhere)

### Arduino panel in web mode
- Arduino panel (sidebar) attempts to connect/detect serial ports on web load
- If serial ports are available → Arduino panel works normally
- If serial ports are unavailable → Arduino panel is hidden (same as `arduino_available=False` in existing template guard)
- No "Requires desktop installation" placeholder shown — simply hidden

### Feature completeness
- All non-hardware features are **fully available** in web mode: analytics, reports, student management, products, transactions history, CSV/Excel exports, email notifications
- Login page is **identical** to desktop — no "Web Portal" badge or branding change
- A **subtle "Web Mode" badge** (small pill) appears in the nav/header area when Arduino is not connected
- Hardware-only sections show a muted placeholder (existing template guard behavior extended)

### Credentials & secrets
- `credentials.json` is uploaded directly to PythonAnywhere via the Files tab; referenced via `GOOGLE_CREDENTIALS_FILE` env var (already configured in `wsgi.py`)
- Secrets (`JWT_SECRET`, `FLASK_SECRET_KEY`, SMTP vars, etc.) are set via a `.env` file on the PythonAnywhere server, loaded by `python-dotenv` at startup
- `web_app.py` retains the **startup guards** for `FLASK_SECRET_KEY` and `JWT_SECRET` — app refuses to start if either is missing/empty
- `wsgi.py` should include **inline comments** documenting required env vars and where `credentials.json` must be placed (as a deployment checklist)

### Claude's Discretion
- Exact visual styling of the "Web Mode" pill badge
- Whether to split `web_app.py` as a separate file from scratch or strip `admin_dashboard.py` in-place
- Exact fallback behavior if serial port detection call itself errors (vs no ports found)
- Loading/detection UX timing for the serial port auto-detect on cashier page

</decisions>
