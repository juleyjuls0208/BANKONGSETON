## Summary

This phase creates `web_app.py` — a stripped, web-safe version of `admin_dashboard.py` — plus updates to `wsgi.py`, `cashier_routes.py`, and `cashier_index.html` to support hardware-absent operation on PythonAnywhere. The primary challenge is that `admin_dashboard.py` has deep hardware coupling: `import serial` at the top level, `socketio.emit()` calls embedded in non-SocketIO routes, and a background thread doing serial reads. All of these must be surgically removed or conditionalized.

The cashier flow requires the most UI work: the existing `cashier_index.html` blocks on `arduinoConnected` to process any sale. Web mode must auto-detect serial ports on load, and if none are found, switch the checkout path to a manual student-lookup modal (search by name or ID → confirm → complete sale). Manual sales are tagged `'Manual'` rather than `'Purchase'`.

PythonAnywhere's standard WSGI hosting does not support background threads or WebSocket upgrades. Flask-SocketIO remains in `web_app.py` for local/desktop use, but its `emit()` calls must be guarded so they do not crash when SocketIO is not active in a WSGI context. The admin Arduino panel is already template-gated by `arduino_available` — passing `False` is the primary mechanism for hiding hardware UI.

**Primary recommendation:** Create `web_app.py` as a copy of `admin_dashboard.py` with targeted removals — conditional `pyserial` import, no startup serial scan, `arduino_available` defaulting to `False`, no background read thread — then patch `cashier_routes.py` with a lookup endpoint and manual-sale path, update `wsgi.py` path and add `.env` loading, and add the manual-mode UI to `cashier_index.html`.

### Pattern 1: Conditional Hardware Import (already established in project)

**What:** `try/except ImportError` wraps hardware libraries; a boolean flag drives feature availability.
**When to use:** Any library that may not be installed or may fail at runtime (pyserial, ArduinoBridge).

```python
# Source: existing pattern in admin_dashboard.py (PHASE3_AVAILABLE, CASHIER_AVAILABLE)
try:
    import serial
    import serial.tools.list_ports
    SERIAL_AVAILABLE = True
except ImportError:
    SERIAL_AVAILABLE = False
```

Then in the `dashboard()` route:
```python
arduino_available = SERIAL_AVAILABLE and bool(serial.tools.list_ports.comports())
```

Or, for `web_app.py` where we always want `False` by default (no runtime detection at app start — detection happens per-request in the cashier):
```python
arduino_available = False  # web_app.py: hardware not expected; cashier detects per-session
```

### Pattern 2: SocketIO emit Guard

**What:** Wrap `socketio.emit()` calls so they are no-ops when called outside a SocketIO request context (e.g., from a regular HTTP route handler in WSGI mode).

**Why needed:** `admin_dashboard.py` calls `socketio.emit()` inside `register_student()`, `report_lost_card()`, etc. — all regular HTTP routes. In WSGI mode without an active SocketIO connection, these calls raise `RuntimeError: Working outside of request context` or silently fail depending on Flask-SocketIO version.

**How to avoid:** Remove `socketio.emit()` calls that fire from non-hardware routes in `web_app.py`. The emit calls in hardware-only routes (`register_student` card scan confirmation, `report_lost_card` card scan) are only reached when Arduino is connected, so they never execute in web mode anyway. The safe approach is to simply remove the emit calls from `web_app.py` — they served as real-time UI feedback for the desktop card-scan UI which is hidden in web mode.

```python
# In web_app.py: remove these emit calls — they only served the Arduino card-scan UI
# socketio.emit('card_registered', {...})   ← REMOVE in web_app.py
# socketio.emit('card_reported_lost', {...})  ← REMOVE in web_app.py
```

**What:** On cashier page load, `loadPorts()` is called. If the response contains serial ports → hardware mode (existing behavior). If ports array is empty or request errors → switch UI to manual mode automatically.

```javascript

        .then(data => {
```

### Pattern 6: wsgi.py `.env` Loading

**What:** Load `.env` before importing the app so all `os.getenv()` calls in `web_app.py` see the secrets.

```python
# wsgi.py — add before the app import:
import os
from dotenv import load_dotenv

# Load secrets from .env file (create this file on PythonAnywhere)
# Required vars: FLASK_SECRET_KEY, JWT_SECRET, GOOGLE_CREDENTIALS_FILE
# Optional: SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD, SMTP_FROM
load_dotenv(os.path.join(os.path.dirname(__file__), '..', '..', '.env'))

from web_app import app as application
```

### Anti-Patterns to Avoid

- **Top-level `import serial`:** Will `ImportError` on PythonAnywhere. Must be `try/except`.
- **`socketio.emit()` from regular HTTP routes:** Fragile in WSGI mode. Remove emit calls that are not inside `@socketio.on()` handlers in `web_app.py`.
- **`from admin_dashboard import ...` in `cashier_routes.py`:** `admin_dashboard` is not the web app module. Must import from `web_app` or use a shared utility module. The safest fix: move `get_sheets_client` and `get_philippines_time` to a shared location, or import lazily from `flask.current_app`.
- **Hardcoded `arduino_available=True` in `web_app.py`:** Must be `False` (or dynamically detected). The existing `admin_dashboard.py` has this hardcoded at line 295; must be changed.
- **Background `read_card_thread()`:** PythonAnywhere standard WSGI does not support background threads. Remove from `web_app.py`. It is only launched when Arduino connects, but the thread registration must not occur at module level.

## Common Pitfalls

### Pitfall 1: `from admin_dashboard import ...` in cashier_routes.py

**What goes wrong:** `cashier_routes.py` lines 159 and 238 import `get_sheets_client` and `get_philippines_time` from `admin_dashboard`. When `web_app.py` is the running module (not `admin_dashboard`), this import either:
- Fails if `admin_dashboard.py` is not importable (e.g., pyserial missing at module level)
- Succeeds but creates a second app instance with duplicated state

**Why it happens:** The cashier blueprint was developed when `admin_dashboard.py` was the only entry point.

**How to avoid:** In `web_app.py`, expose `get_sheets_client` and `get_philippines_time` as module-level symbols (they already exist in the file — just ensure they are accessible). Then update `cashier_routes.py` to import from `web_app` instead. Or better: move these utilities to a shared module (e.g., `backend/utils_sheets.py`) to avoid circular imports.

**Recommended fix for this phase:** Import lazily inside the route functions using `current_app`:
```python
# cashier_routes.py — replace module-level imports:
# from admin_dashboard import get_sheets_client  ← REMOVE
# Use instead:
from flask import current_app
sheets_client = current_app.get_sheets_client()  # if attached to app
# OR import from web_app at function scope only:
```
Simplest viable fix: in `cashier_routes.py`, change the import to try `web_app` with fallback to `admin_dashboard`:
```python
try:
    from web_app import get_sheets_client, get_philippines_time
except ImportError:
    from admin_dashboard import get_sheets_client, get_philippines_time
```

### Pitfall 2: wsgi.py Wrong Path

**What goes wrong:** Current `wsgi.py` has `project_home = '/home/bankoseton/FinanceDashboard'` — this path does not exist on PythonAnywhere (project is at `BANKONGSETON`). App fails to start with `ModuleNotFoundError`.

**How to avoid:** Update line 8 to `/home/bankoseton/BANKONGSETON/backend/dashboard`.

### Pitfall 3: .env Not Loaded Before App Import

**What goes wrong:** `web_app.py` startup guards (`FLASK_SECRET_KEY`, `JWT_SECRET`) run at import time. If `load_dotenv()` is not called before `from web_app import app`, the guards fire and the app refuses to start.

**How to avoid:** `load_dotenv()` must be the first statement in `wsgi.py`, before any Flask app import.

### Pitfall 4: Manual Mode Checkout Not Tagging 'Manual'

**What goes wrong:** Existing `complete_sale` always writes `'Purchase'` as transaction type. Manual sales would be indistinguishable from RFID sales in reports.

**How to avoid:** `complete_sale` must check for `manual_student_id` in the request body and use `'Manual'` as transaction type when present.

### Pitfall 5: `checkout()` Guard in cashier_index.html

**What goes wrong:** Current `checkout()` function (line 305) has `if (!arduinoConnected) { alert('...'); return; }` — this blocks all sales in web mode. The manual-mode path must replace this guard rather than just remove it.

**How to avoid:** Change the guard to a branch: if `arduinoConnected` → existing RFID flow; else → open manual lookup modal.

### Pitfall 6: initWebSocket() SocketIO Connection in WSGI

**What goes wrong:** `cashier_index.html` calls `io()` (SocketIO connect) on page load. On PythonAnywhere standard WSGI, the WebSocket upgrade is rejected, causing console errors (but not necessarily breaking the page).

**How to avoid:** Wrap `initWebSocket()` so it is only called if hardware mode is active. In manual mode, skip the SocketIO connection entirely.

### Pitfall 7: GOOGLE_CREDENTIALS_FILE Relative Path

**What goes wrong:** Current `wsgi.py` sets `os.environ['GOOGLE_CREDENTIALS_FILE'] = 'credentials.json'` — relative path. On PythonAnywhere, the working directory may not be the project root.

**How to avoid:** Use an absolute path:
```python
os.environ.setdefault('GOOGLE_CREDENTIALS_FILE',
    os.path.join(os.path.dirname(__file__), 'config', 'credentials.json'))
```
