### Task 2 — iOS: Add `cardStatus` field to `Student` model and fix `fatalError()` (`LoginModels.swift` + `APIClient.swift`)
**Requirements:** REQ-BUG-MOB-01 (model), REQ-BUG-MOB-03 (fatalError)
**Files:**
- `mobile/ios/BankongSetonStudent/Models/LoginModels.swift`
- `mobile/ios/BankongSetonStudent/Core/Network/APIClient.swift`

**Actions (LoginModels.swift):**
1. Open `LoginModels.swift`. Locate the `Student` struct.
2. Add a `cardStatus: String?` field with a custom `CodingKeys` mapping to `"status"`:
   ```swift
   struct Student: Codable {
       let studentId: String
       let name: String
       let cardStatus: String?
       enum CodingKeys: String, CodingKey {
           case studentId = "student_id"
           case name
           case cardStatus = "status"
       }
   }
   ```
   - Must be `String?` (Optional) so existing API responses that omit `"status"` don't cause decoding failures.
   - `CodingKeys` enum must also include cases for `studentId` and `name` with their existing mappings to avoid a compiler error (custom `CodingKeys` requires ALL synthesized keys to be listed).

**Actions (APIClient.swift):**
1. Open `APIClient.swift`. Locate the `APIError` enum (lines 1–10).
2. Add `case invalidURL` as the first case:
   ```swift
   enum APIError: Error {

       case unauthorized
       case httpError(Int)
       case decodingError(Error)
       case networkError(Error)
   }
   ```
3. Locate the `fatalError` in `authenticatedRequest()` (line ~27):
   ```swift
   // OLD:
   fatalError("Invalid URL for path: \(path)")
   // NEW:
   throw APIError.invalidURL
   ```
4. Locate the `fatalError` in `login()` (line ~70):
   ```swift
   // OLD:
   fatalError("Invalid login URL")
   // NEW:
   throw APIError.invalidURL
   ```

**Verification:**
- `Student` struct has `cardStatus: String?` with `CodingKeys` that maps it to `"status"`.
- `APIError` enum has `case invalidURL` as a case.
- Neither `authenticatedRequest()` nor `login()` contain the word `fatalError`.

### Task 3 — Android: Fix `isCardLost` clearance on login (`LoginActivity.kt`)
**Requirement:** REQ-BUG-MOB-02
**File:** `mobile/student_app_v2/app/src/main/java/com/bankongseton/student/LoginActivity.kt`

**Background (from research):** `SecureStorage.clearAuth()` already preserves `isCardLost` — the flag
is never deleted on logout. The bug is on the login side: after a successful login where the backend
confirms the card is active, the app never clears the flag. The `Student` model already has
`val status: String` — no model changes needed.

**Actions:**
1. Open `LoginActivity.kt`. Locate the `performLogin()` method and its success callback.
2. Inside the `if (response.isSuccessful)` block, after calling `secureStorage.saveAuthToken(...)` and
   `secureStorage.saveStudentId(...)`, add:
   ```kotlin
   // Clear lost card flag if backend confirms card is active
   if (loginResponse.student.status != "lost") {
       secureStorage.setCardLost(false)
   }
   ```
3. The `secureStorage` instance must already be declared in `LoginActivity` — do NOT add a new
   `SecureStorage` instantiation if one already exists.

**Verification:**
- `performLogin()` response handler calls `secureStorage.setCardLost(false)` when `student.status != "lost"`.
- No changes to `SecureStorage.kt`, `Models.kt`, or any other file.

#### Step 5a — `BankoHceService.kt`: Make fields private, add controlled API

1. Locate the `companion object` in `BankoHceService.kt`. Find:
   ```kotlin
   @Volatile var currentToken: String? = null
   @Volatile var isPaymentAuthorized: Boolean = false
   ```
2. Make both fields `private`:
   ```kotlin
   @Volatile private var currentToken: String? = null
   @Volatile private var isPaymentAuthorized: Boolean = false
   ```
   Keep `@Volatile` on both. Thread safety is required.

3. Add three methods to the companion object immediately after the private fields:
   ```kotlin
   fun authorize(token: String) {
       currentToken = token
       isPaymentAuthorized = true
   }

   fun reauthorize() {
       check(currentToken != null) { "reauthorize() called with no token loaded" }
       isPaymentAuthorized = true
   }

   fun deauthorize() {
       currentToken = null
       isPaymentAuthorized = false
   }
   ```
   - `authorize(token)` — full authorization with a new token
   - `reauthorize()` — sets `isPaymentAuthorized = true` when token is already loaded (used by overlay)
   - `deauthorize()` — clears both fields (used for logout/cancel/timeout)

4. **Internal reads** (`processCommandApdu()`, `handleSelectCommand()`, `onDeactivated()`) access the
   fields directly — they are inside `BankoHceService` class so private fields remain accessible.
   Do NOT change any internal read or write of these fields within `BankoHceService.kt` itself.
