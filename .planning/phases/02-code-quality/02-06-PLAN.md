<task type="auto">
  <name>Task 1: Fix get_logger() to return bangko.* child loggers</name>
  <files>backend/errors.py</files>
  <action>
    Edit `get_logger()` in `backend/errors.py` (currently line 107–109):

    **Current code:**
    ```python
    def get_logger(name: str = 'bangko') -> logging.Logger:
        """Get logger instance"""
        return logging.getLogger(name)
    ```

    **Replace with:**
    ```python
    def get_logger(name: str = 'bangko') -> logging.Logger:
        """Get a logger in the bangko.* hierarchy.

        If name is 'bangko' or already starts with 'bangko.', returns the logger
        as-is.  Otherwise, prepends 'bangko.' so callers using get_logger(__name__)
        (e.g. 'admin_dashboard') automatically become children of the bangko logger
        and inherit its StreamHandler without needing their own handler.
        """
        if name == 'bangko' or name.startswith('bangko.'):
            return logging.getLogger(name)
        return logging.getLogger(f'bangko.{name}')
    ```

    **Why:** The `bangko` logger has a `StreamHandler` attached by `setup_logging()` with
    `propagate=False`. Callers using `get_logger(__name__)` currently receive root-level
    loggers (e.g. `logging.getLogger('admin_dashboard')`) that never propagate through the
    `bangko` hierarchy — so their messages are silently dropped. Making them children
    (e.g. `bangko.admin_dashboard`) means they inherit the handler automatically.

    **Note:** The STATE.md decision "[02-02]: Kept get_logger() unchanged for downstream
    compatibility" was recorded at plan time to defer this change. The VERIFICATION.md
    confirms the gap is real and must be fixed. The new implementation is backward-compatible:
    callers that already pass 'bangko' explicitly still get the root bangko logger.
  </action>
  <verify>

    ```python
        ...
    ```

    **File 2: `backend/api/api_server.py`**

    The `__main__` block starts at line 780. `setup_logging` is NOT currently imported in
    api_server.py — only `get_logger` is imported (line 24: `from errors import get_logger`).

    First, update the import at the top of the try/except block (line 24) to also import
    `setup_logging`:

    ```python
    from errors import setup_logging, get_logger
    ```

    Then add `setup_logging()` as the FIRST line inside `if __name__ == '__main__':`:

    ```python
    if __name__ == '__main__':
        setup_logging()  # activate bangko StreamHandler before first log call
        port = int(os.getenv('API_PORT', 5001))
        ...
    ```

    **Why:** Without `setup_logging()`, the `bangko` logger has no handlers when `logger.info()`
    is first called. Python's "no handlers found" behavior silently drops the message (or in
    Python ≥ 3.2 emits a `logging.lastResort` WARNING-level stderr fallback at best).
    Calling `setup_logging()` first ensures the StreamHandler is active before any log line fires.

    **Note:** `setup_logging()` uses `logger.handlers.clear()` before adding its handler, so
    it is safe to call even if Flask/SocketIO later configures its own logging — the bangko
    logger is isolated by `propagate=False` and won't interfere with werkzeug logging.
  </action>
