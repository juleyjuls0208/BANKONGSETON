**LoginView.swift — Two fixes (REQ-UX-04 + REQ-UX-05):**

Fix A — PIN content type:
Find the `SecureField` for the PIN input. Change its content type modifier:
```swift
// Before:
.textContentType(.password)
// After:
.textContentType(.oneTimeCode)
```
`.oneTimeCode` suppresses the password autofill suggestion bar for numeric PIN fields.

Fix B — Keyboard avoidance on iPhone SE:
The current form content is in a `VStack` inside a `NavigationStack`. Wrap the `VStack` in a `ScrollView`:
```swift
// Before:
NavigationStack {
    VStack(spacing: 24) {
        // ...logo, fields, button, Spacer()...
    }
}

// After:
NavigationStack {
    ScrollView {
        VStack(spacing: 24) {
            // ...same content...
            Spacer(minLength: 20)
        }
    }
}
```
Replace the trailing `Spacer()` (full-height spacer, only useful in fixed-frame VStack) with `Spacer(minLength: 20)` inside the ScrollView to give some bottom breathing room. The ScrollView lets content scroll above the keyboard so the Sign In button stays visible.

**BudgetView.swift — hasEdited guard (REQ-UX-01):**

Make four changes to prevent the server response from overwriting text the user is actively typing:

1. Add `@State` property alongside existing `@State` declarations:
```swift
@State private var hasEdited = false
```

2. In the `.task` block, guard the `limitInput` assignment:
```swift
// Before (inside .task):
limitInput = String(format: "%.2f", viewModel.limit)

// After:
if !hasEdited {
    limitInput = String(format: "%.2f", viewModel.limit)
}
```

3. On the TextField for the budget limit, add an `onChange`:
```swift
.onChange(of: limitInput) { _ in hasEdited = true }
```
**IMPORTANT — iOS 16 compatibility:** Use the single-argument closure `{ _ in }` form. Do NOT use the iOS 17 two-argument form `{ oldValue, newValue in }`. This project targets iOS 16.

4. On the outermost container (e.g., `NavigationStack` or `Form`), add `onDisappear` to reset the flag:
```swift
.onDisappear { hasEdited = false }
```
Resetting on disappear ensures the server value reloads fresh on the next visit. Unsaved typed input is intentionally discarded on navigation away.

**TransactionsView.swift — empty state overlay (REQ-UX-02):**

Add a new `.overlay` block after the existing loading/error overlays:
```swift
.overlay {
    if viewModel.transactions.isEmpty && !viewModel.isLoading && viewModel.errorMessage == nil {
        VStack(spacing: 8) {
            Text("No transactions yet")
                .font(.headline)
                .foregroundColor(.primary)
            Text("Your transaction history will appear here.")
                .font(.subheadline)
                .foregroundColor(.secondary)
                .multilineTextAlignment(.center)
        }
        .padding()
    }
}
```
Placement: After the existing `.overlay { if viewModel.isLoading { ... } }` block.
The three-condition guard (`isEmpty && !isLoading && errorMessage == nil`) ensures the empty state only shows when the list is genuinely empty — not while loading, and not when an error is being displayed instead.
