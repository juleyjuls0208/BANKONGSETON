In `NfcManager.kt`, make three targeted changes:

**1. Add `hashPin()` private method** (place near top of the class body, after companion object constants):
```kotlin
private fun hashPin(pin: String): String {
    val digest = java.security.MessageDigest.getInstance("SHA-256")
    val hashBytes = digest.digest(pin.toByteArray(Charsets.UTF_8))
    return hashBytes.joinToString("") { "%02x".format(it) }
}
```
`java.security.MessageDigest` is in the JDK — no import needed if using fully-qualified name.

**2. Update `setPin()`** — find the line with `putString(KEY_NFC_PIN, pin)` and change to:
```kotlin
securePrefs.edit().putString(KEY_NFC_PIN, hashPin(pin)).apply()
```

**3. Update `verifyPin()`** — replace the comparison logic with a migration-aware version:
```kotlin
fun verifyPin(enteredPin: String): Boolean {
    val stored = securePrefs.getString(KEY_NFC_PIN, null) ?: return false
    // Migrate legacy plaintext PIN on first successful verify
    if (stored == enteredPin) {
        setPin(enteredPin)  // re-save as hash
        return true
    }
    return stored == hashPin(enteredPin)
}
```
The migration check (`stored == enteredPin`) handles existing users who stored plaintext before this fix. On first successful verify, their PIN is silently re-saved as a hash. After that, `stored == enteredPin` will always be false (hash ≠ short PIN string), so the hash path runs.

Do NOT change the `registerDevice()` call that sends the PIN to the backend — that's a separate API-level concern flagged as out of scope for this phase.
