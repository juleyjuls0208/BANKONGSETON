# Phase 20.1: Arduino PN532 NFC + Backend Integration + Student App Payment + Firmware Hardening - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Complete the end-to-end NFC phone tap payment flow. The Android HCE side (`BankoHceService.kt`) and Flask backend (`/api/nfc/pay`) are already built. The gap is that the Arduino firmware can only read UIDs — it cannot perform APDU exchange. This phase adds APDU capability to the R3 (PN532 over SPI), upgrades the R4 from MFRC522 to PN532, wires the extracted HCE token through the serial/WiFi paths to `/api/nfc/pay`, and updates the cashier UI to handle the new NFC payment event.

</domain>

<decisions>
## Implementation Decisions

### APDU Exchange Strategy (R3 PN532)
- **Detection:** Attempt SELECT APDU first on every tap. If APDU succeeds → HCE phone path. If APDU fails or times out → treat as physical RFID card, fall back to UID path. Both modes active simultaneously.
- **Timeout:** 500ms for APDU exchange attempt before falling back to UID path.
- **Exchange depth:** Single exchange only — Arduino sends SELECT AID, phone responds with 48-char token + SW_OK (`0x90 0x00`). No further APDU commands after token is received.
- **AID:** `F0 42 41 4E 4B 4F 4E 47 53 45 54 4F 4E` (matches `BankoHceService.kt` exactly).
- **Security:** Planner decides whether to add a second GET DATA exchange for amount confirmation, based on security requirements review.

### R4 Hardware Upgrade
- Replace MFRC522 with PN532 on the R4 (UNO R4 WiFi). MFRC522 does not support ISO 14443-4 APDU.
- This phase specifies the SPI pin mapping and provides an updated `secrets.h` template with PN532 config. Planner writes the actual R4 firmware.
- The R4 has WiFi — it POSTs the extracted token directly to `/api/nfc/pay` (not via serial).

### Token Delivery — R3 (Serial Path)
- New serial line format: `NFC|<48-char-token>` (analogous to existing `CARD|<uid>` format).
- `ArduinoBridge` in `arduino_bridge.py` is updated to parse `NFC|` lines and emit a new `nfc_payment` SocketIO event.
- Existing `card_read` SocketIO event is preserved unchanged for RFID UID taps.
- On APDU error: Arduino sends `ERROR|NFC_FAIL` over serial. ArduinoBridge emits a socket event for this too.

### Token Delivery — R4 (WiFi Path)
- R4 POST directly to `/api/nfc/pay` with `{ virtual_card_token, items, total }` after APDU exchange succeeds.
- R4 does not use the serial path.

### Backend NFC Payment Call
- `ArduinoBridge` (running inside the `web_app.py` process) calls `/api/nfc/pay` on `api_server.py` internally (HTTP call) when it receives an `NFC|` serial line.
- `items` and `total` come from the cashier UI state (pre-selected by cashier before tap).
- The existing `/api/nfc/pay` endpoint in `api_server.py` is used as-is — no new endpoints needed.
- Replay protection is entirely the backend's responsibility. No Arduino-side token caching.

### Cashier UI — NFC Payment Flow
- Item selection flow unchanged: cashier pre-selects items first, then student taps phone. Same sequence as existing RFID card flow.
- When `nfc_payment` socket event arrives, cashier UI shows the same payment confirmation modal as RFID, but with:
  - NFC/phone icon badge
  - Blue color accent (vs green for RFID)
- On token lookup failure (token not found, card deactivated, insufficient balance): show error in the existing payment result area with a retry button. No full-screen overlay.

### Firmware Hardening — R3 LCD States
- LCD shows step-by-step states during APDU flow:
  1. `Tap Phone...` — waiting for tap
  2. `Reading...` — during APDU exchange
  3. `OK` — successful token extraction
  4. `NFC Error` — on APDU failure
- Reuses existing LCD update pattern already in the R3 sketch.

### Firmware Hardening — Error Handling
- APDU failure: send `ERROR|NFC_FAIL` over serial, show `NFC Error` on LCD, reset and wait for next tap. No retries on APDU failure — single attempt per tap.
- RFID UID fallback happens only on APDU timeout (500ms), not on explicit error SW bytes (those are a definitive failure, not a "not an HCE phone" signal).

### Claude's Discretion
- Whether to add a second GET DATA APDU exchange for amount confirmation (planner decides after reviewing security requirements).
- Exact SPI pin mapping for PN532 on R4 (planner specifies based on R4 pinout).
- Internal HTTP call mechanism ArduinoBridge uses to reach `api_server.py` (direct requests, or via the existing inter-process patterns in the codebase).

</decisions>

<specifics>
## Specific Ideas

- The HCE SELECT AID must exactly match what `BankoHceService.kt` expects: `F0 42 41 4E 4B 4F 4E 47 53 45 54 4F 4E`.
- The 48-char ASCII token format is already defined by `BankoHceService.kt` — Arduino just reads it as a byte string and forwards it verbatim.
- Blue for NFC, green for RFID — cashier UI color distinction is intentional for quick visual scanning by cashier staff.
- "Same as RFID flow" for item selection — cashier workflow muscle memory should not change.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ArduinoBridge` (`backend/dashboard/arduino_bridge.py`): Already parses `<CARD|UID>` serial lines and emits `card_read` SocketIO events. Extend with `NFC|<token>` parser and `nfc_payment` event.
- `NFCService.get_virtual_card_by_token()` (`backend/nfc_payments.py`): Token lookup against VirtualCards Google Sheet. Used by `/api/nfc/pay` — no changes needed.
- `/api/nfc/pay` (`backend/api/api_server.py`): Already accepts `{ virtual_card_token, items, total }`. Already returns payment result. No changes needed.
- `/api/arduino/card-read` (`backend/dashboard/web_app.py`): WiFi endpoint for R4 RFID UIDs with `ARDUINO_API_KEY` auth pattern — R4 NFC WiFi POST can follow the same auth pattern.
- `bankongseton_nfc_r3.ino` (`arduino/bankongseton_nfc_r3/`): PN532 over SPI already initialized. `readPassiveTargetID()` already called. LCD update pattern already in place. APDU exchange is the new addition.

### Established Patterns
- Serial format `<TYPE|DATA>` with angle brackets: existing pattern for R3. New `NFC|` format follows this convention.
- `ARDUINO_API_KEY` header auth for WiFi endpoints: R4 token POST should use the same auth.
- `secrets.h` pattern (Phase 18): WiFi credentials and API keys in `secrets.h`, not hardcoded. R4 PN532 config goes in the same file.
- SocketIO for cashier UI updates: all Arduino events reach the cashier UI via SocketIO. `nfc_payment` event follows this pattern.

### Integration Points
- R3 firmware → `ArduinoBridge` serial parser → internal HTTP call → `/api/nfc/pay` → `NFCService` → Google Sheet
- R4 firmware → WiFi POST → `/api/nfc/pay` → `NFCService` → Google Sheet
- `nfc_payment` SocketIO event → cashier UI → payment confirmation modal
- `ERROR|NFC_FAIL` serial line → `ArduinoBridge` → cashier UI error state

</code_context>
