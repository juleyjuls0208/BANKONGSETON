<objective>
Implement NFC APDU firmware on two Arduino boards: R3 (reader/LCD) adds dual-mode APDU + UID-fallback to the existing PN532 sketch; R4 (WiFi poster) replaces MFRC522 with PN532 and changes the POST target to `/api/nfc/pay`.

Purpose: Enable tap-to-pay with student phones via HCE. R3 extracts a 48-char cryptographic token from the phone's NDEF app; R4 forwards it to the backend payment endpoint.

Output:
- `bankongseton_nfc_r3.ino` — dual-mode APDU/UID firmware with correct Serial format and LCD states
- `bankongseton_rfid.ino` — PN532 WiFi firmware posting JSON `{"token":"..."}` to `/api/nfc/pay`
</objective>

     ```cpp
     uint8_t uid[7];
     uint8_t uidLength;
     // readPassiveTargetID blocks until card found; wrap in a non-blocking pattern
     // For R4 (WiFi poster), a simple blocking read is acceptable since R3 handles timing
     if (nfc.readPassiveTargetID(PN532_MIFARE_ISO14443A, uid, &uidLength, 1000)) {
       // Build UID hex string for httpPost compatibility
       String uidStr = "";
       for (uint8_t i = 0; i < uidLength; i++) {
         if (uid[i] < 0x10) uidStr += "0";
         uidStr += String(uid[i], HEX);
       }
       deliverUID(uidStr);
     }
     ```

   - Keep all retry logic, `MAX_RETRIES`, `connectWiFi()`, `ensureWiFi()` unchanged

6. **Serial fallback** in deliverUID: change `Serial.print("CARD|")` to `Serial.print("NFC|")` so the bridge can handle it. Actually — leave as `CARD|` since the bridge already handles both. Do NOT change serial format in R4.
