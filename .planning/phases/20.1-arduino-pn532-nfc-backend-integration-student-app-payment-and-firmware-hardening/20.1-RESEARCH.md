Phase 20.1 completes the end-to-end NFC phone-tap payment flow across four work areas: R3 Arduino firmware (add APDU exchange alongside existing UID path), R4 Arduino firmware (swap MFRC522 for PN532 and POST token directly), ArduinoBridge Python backend (parse `NFC|<token>` serial lines and forward via HTTP + SocketIO), and Cashier UI (handle `nfc_payment` event with blue modal variant). No new dependencies are required in any layer.

**Primary recommendation:** Add APDU path in firmware as a pre-check before UID fallback (dual-mode); use `inDataExchange()` only after card detection via `readPassiveTargetID(timeout=500)`; emit `NFC|<token>` over Serial; bridge picks it up as a new `elif` branch and POSTs internally.

## Architecture Patterns

### R3 Firmware тАФ Dual-Mode APDU + UID Fallback

```
loop():
  1. readPassiveTargetID(PN532_MIFARE_ISO14443A, uid, &uidLen, 500)
     тЖТ timeout 500ms
     тЖТ if false: continue (no card)
  2. Card detected тЖТ attempt APDU SELECT AID via inDataExchange()
     тЖТ if success && SW=9000 && responseLength==50:
         Serial.print("NFC|")
         Serial.println(token_as_string)  // 48 chars
     тЖТ else (timeout / bad SW / wrong length):
         Serial.print("CARD|")
         Serial.println(uid_as_hex)       // legacy UID path
  3. Debounce delay
```

**Key insight:** `readPassiveTargetID()` with timeout parameter is the safe card-detection gate. Only call `inDataExchange()` AFTER this returns `true` тАФ the card is then listed internally and `_inListedTag` is set.

### R4 Firmware тАФ PN532 + Direct WiFi POST

```
loop():
  1. readPassiveTargetID(PN532_MIFARE_ISO14443A, uid, &uidLen, 500)
  2. if card present:
       attempt APDU тЖТ get token
       httpPost("/api/nfc/pay", body="{\"virtual_card_token\":\"<48chars>\"}")
       OR fallback: httpPost("/api/arduino/card-read", body=uid)
  3. Debounce
```

Reuse existing `httpPost(String path, String body)` function verbatim тАФ only change the path string and body payload.

## Adafruit PN532 APDU API

All signatures confirmed from `Adafruit_PN532.h` and `Adafruit_PN532.cpp` (GitHub).

### Constructor (SPI mode)
```cpp
// Use the single-pin SS constructor тАФ hardware SPI
#define PN532_SS 10
Adafruit_PN532 nfc(PN532_SS);   // already in R3 sketch тАФ reuse verbatim
```

### `readPassiveTargetID` тАФ Card Detection with Timeout
```cpp
bool readPassiveTargetID(
    uint8_t cardbaudrate,   // PN532_MIFARE_ISO14443A = 0x00
    uint8_t *uid,           // out: UID bytes
    uint8_t *uidLength,     // out: UID length (4 or 7)
    uint16_t timeout        // ms; 0 = wait forever; use 500
);
// Returns: true if card found within timeout, false if timeout
// SIDE EFFECT: sets internal _inListedTag тАФ required before inDataExchange()
```

### `inDataExchange` тАФ Raw APDU Exchange
```cpp
bool inDataExchange(
    uint8_t *send,              // APDU command bytes (your SELECT AID array)
    uint8_t sendLength,         // length of send buffer
    uint8_t *response,          // out: response bytes (allocate тЙе64)
    uint8_t *responseLength     // in: max buffer size; out: actual bytes received
);
// Returns: true on successful exchange (does NOT check SW bytes тАФ you must check)
// Internal timeout: 1000ms (hardcoded in sendCommandCheckAck)
// Prepends 0x40 (INDATAEXCHANGE) and _inListedTag automatically
// Max response buffer: 64 bytes (pn532_packetbuffer size)
```

### тЪая╕П Critical Ordering Constraint
```
readPassiveTargetID()  тЖТ  sets _inListedTag
       тЖУ
inDataExchange()       тЖТ  uses _inListedTag internally
```
**Never call `inDataExchange()` without a preceding successful `readPassiveTargetID()`.**
The `inListPassiveTarget()` raw method has a hardcoded 30-second blocking wait тАФ do NOT use it directly.

## APDU Command Construction

### SELECT AID Command (19 bytes)

```cpp
uint8_t selectAID[] = {
    0x00,                         // CLA
    0xA4,                         // INS: SELECT
    0x04,                         // P1: select by AID
    0x00,                         // P2
    0x0D,                         // Lc: AID length = 13 bytes
    0xF0, 0x42, 0x41, 0x4E,      // AID: F0 BANK
    0x4B, 0x4F, 0x4E, 0x47,      // AID: KONG
    0x53, 0x45, 0x54, 0x4F, 0x4E, // AID: SETON
    0x00                          // Le: return all
};
uint8_t selectAIDLen = 19;
```

### Response Parsing

```cpp
uint8_t response[64];
uint8_t responseLength = sizeof(response);

if (nfc.inDataExchange(selectAID, selectAIDLen, response, &responseLength)) {
    // Check SW1=0x90 SW2=0x00
    if (responseLength == 50 &&
        response[48] == 0x90 &&
        response[49] == 0x00) {

        // Extract 48-char ASCII token
        char token[49];
        memcpy(token, response, 48);
        token[48] = '\0';

        Serial.print("NFC|");
        Serial.println(token);   // emits: NFC|<48 chars>\r\n

    } else {
        // Wrong SW or wrong length тЖТ UID fallback
        goto uid_fallback;
    }
} else {
    // inDataExchange returned false тЖТ UID fallback
    goto uid_fallback;
}
```

> **Note:** `goto uid_fallback` is a C pattern for clarity; in practice use a boolean flag or restructure as if/else.

### Serial Output Format

| Scenario | Serial output |
|----------|--------------|
| HCE phone tap (success) | `NFC|ABCDEF...48chars\r\n` |
| Physical MIFARE/UID card | `CARD|A1B2C3D4\r\n` |
| APDU exchange error | `ERROR|NFC_FAIL\r\n` (optional тАФ useful for debug) |

## R4 Pin Mapping

Confirmed from R3 sketch (`#define PN532_SS 10`) + Arduino UNO R4 WiFi hardware documentation.

| Signal | Arduino Pin | Notes |
|--------|-------------|-------|
| MOSI | D11 | Hardware SPI тАФ same as R3 |
| MISO | D12 | Hardware SPI тАФ same as R3 |
| SCK | D13 | Hardware SPI тАФ same as R3 |
| SS (CS) | D10 | `#define PN532_SS 10` |
| VCC | 3.3V or 5V | PN532 module accepts both; R4 GPIO is 5V tolerant |
| GND | GND | тАФ |
| SEL0 | GND (LOW) | Selects SPI mode |
| SEL1 | VCC (HIGH) | Selects SPI mode |

**R4-specific note:** The UNO R4 WiFi uses the RA4M1 MCU. Its hardware SPI pins are physically identical to UNO R3 (D11/D12/D13/D10). The existing `WiFiS3.h` library and `SPI.h` coexist without conflict тАФ confirmed by existing R4 sketch which already uses both.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| APDU framing / TFI header | Custom SPI byte writes | `inDataExchange()` | Library handles INDATAEXCHANGE opcode, `_inListedTag` injection, ACK handshake, 1000ms timeout |
| Card detection with timeout | Polling loop with `millis()` | `readPassiveTargetID(..., timeout)` | Built-in timeout parameter; also sets internal tag index as side effect |
| PN532 SPI initialization | Raw SPI register writes | `nfc.begin()` + `nfc.SAMConfig()` | SAMConfig enables normal mode; skipping it causes silent failures |
| HTTP client on R4 | Raw TCP socket writes | Existing `httpPost()` in R4 sketch | Already handles WiFiClient, headers, response reading |
| JSON body on R4 | String concatenation | Simple string concat is fine for a single string field | Only add ArduinoJson if you need >2 fields |
| SocketIO emit patterns | Custom WebSocket frames | `socketio.emit()` | Already wired up; same as `card_read` |

**Key insight:** The Adafruit library abstracts all PN532 framing complexity. The only custom code needed is the 19-byte SELECT AID array and the SW byte check тАФ everything else is a library call.

## Common Pitfalls

### Pitfall 1: Calling `inDataExchange()` Without Prior `readPassiveTargetID()`
**What goes wrong:** `inDataExchange()` sends to `_inListedTag=0` (uninitialised) тЖТ PN532 returns error or garbage.
**Why it happens:** Developers assume `inDataExchange()` auto-detects the card.
**How to avoid:** Always gate on `readPassiveTargetID()` returning `true` first. The dual-mode pattern above does this correctly.
**Warning signs:** `inDataExchange()` returns `false` consistently even with a phone present.

### Pitfall 2: `inListPassiveTarget()` 30-Second Blocking Wait
**What goes wrong:** Calling `inListPassiveTarget()` directly blocks the loop for 30 seconds if no card is present.
**Why it happens:** The internal `waitready` call has a hardcoded 30s timeout in the library source.
**How to avoid:** **Never call `inListPassiveTarget()` directly.** Use `readPassiveTargetID()` with the `timeout` parameter.

### Pitfall 3: PN532 SPI Mode Not Selected (SEL pins)
**What goes wrong:** PN532 starts in UART or I2C mode; SPI reads return 0xFF or hang.
**Why it happens:** PN532 mode is hardware-selected by SEL0/SEL1 pins on the module.
**How to avoid:** Wire SEL0=GND, SEL1=VCC before powering on. Verify with `nfc.getFirmwareVersion()` in `setup()`.

### Pitfall 4: APDU Response Shorter Than Expected
**What goes wrong:** Token extraction reads garbage because phone sent partial response.
**Why it happens:** HCE app returned fewer than 48 bytes (e.g., app bug or SELECT AID not matched).
**How to avoid:** Always check `responseLength == 50` AND `SW == 0x9000` before extracting token. Fall through to UID path otherwise.

### Pitfall 5: Serial `line.strip()` Missing in ArduinoBridge
**What goes wrong:** `line.startswith("NFC|")` matches but token has trailing `\r\n`, making `len(token) == 50` instead of 48.
**Why it happens:** Arduino `Serial.println()` appends `\r\n`; Python `readline()` may or may not strip it.
**How to avoid:** Always call `.strip()` on the token after slicing: `token = line[4:].strip()`.

### Pitfall 6: Port 5001 Assumption
**What goes wrong:** Internal `requests.post` to wrong port; 404 or connection refused.
**Why it happens:** Port is not explicitly documented тАФ inferred from CORS config.
**How to avoid:** Read port from environment variable or shared config constant rather than hardcoding `5001`. Verify by checking `web_app.py` CORS origins at startup.

### Pitfall 7: Missing `SAMConfig()` Call
**What goes wrong:** `nfc.begin()` alone is insufficient; PN532 stays in power-down mode.
**Why it happens:** `SAMConfig()` must be called to switch PN532 to normal operating mode.
**How to avoid:** Always call both: `nfc.begin(); nfc.SAMConfig();` тАФ the R3 sketch already does this correctly.
