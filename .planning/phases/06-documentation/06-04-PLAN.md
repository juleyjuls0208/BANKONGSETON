**Structure:**

**H1: NFC Integration Guide**

**H2: Overview**
BankongSeton supports NFC payments via Android Host Card Emulation (HCE). The backend NFC API is implemented and ready (Phase 5). This guide describes what the Android app must implement to enable students to pay by tapping their phone instead of their physical RFID card.

NFC v2 status: The backend API is complete. The Android app does not yet implement NFC. This guide is the specification for implementing it.

**H2: How It Works**
High-level flow:
1. Student registers phone as virtual card via `POST /api/nfc/register` → receives virtual_card_token + device_token
2. Phone stores both tokens securely
3. At payment terminal, phone emulates an NFC card using HCE
4. Terminal reads the APDU response containing virtual_card_token and device_token
5. Cashier system calls `POST /api/nfc/pay` with the tokens
6. Backend validates, deducts balance, logs transaction (TransactionType='NFC Purchase')

**H2: NFC Registration**
Endpoint: `POST /api/nfc/register`
- Auth: Session Bearer token (student must be logged in)
- Request: `{}` (no body needed; student identity from session)
- Response:
```json
{
  "virtual_card_token": "uuid-or-token-string",
  "device_token": "device-specific-token"
}
```
- What it creates in Google Sheets: new row in VirtualCards sheet (StudentID, VirtualCardToken, DeviceToken, MoneyCardNumber, CreatedAt, IsActive)
- Store both tokens securely on the device (EncryptedSharedPreferences)

**H2: HCE Implementation**

**H3: Application Identifier (AID)**
The Android app must declare the AID that the cashier terminal will select:
```
AID: F049494F4E41
```
This is registered in `AndroidManifest.xml` and the HCE service.

```xml
    </aid-group>

```

**H2: APDU Response Format**
When the terminal sends SELECT AID, the HCE service responds with:
```
Bytes: virtual_card_token|device_token
Status word: 0x9000 (success)
```
The `|` character is the literal separator between virtual_card_token and device_token.
On error: respond with `0x6F00`.

**H2: NFC Payment Flow**
The cashier terminal (or a future NFC-capable cashier) calls:
- Endpoint: `POST /api/nfc/pay`
- Auth: JWT Bearer token (cashier role) + `X-Device-Token: <device_token>` header
- Request:
```json
{
  "virtual_card_token": "uuid-or-token",
  "items": [{"name": "Rice", "price": 25.0, "quantity": 1}],
  "total": 25.0
}
```
- Response on success:
```json
{
  "success": true,
  "new_balance": 225.0,
  "transaction_id": "..."
}
```
- Transaction logged with TransactionType='NFC Purchase' (distinct from 'Purchase' so students can filter in transaction history)

Recreate the sequence diagram from the existing nfc-integration-guide.md showing the full flow: Student Phone → NFC Antenna → Cashier Terminal → POST /api/nfc/pay → Backend → Google Sheets.

**H2: Security Considerations**
- Virtual card token: UUID, server-generated, stored in VirtualCards sheet
- Device token: device-specific, passed via X-Device-Token header and stored in VirtualCards sheet
- IsActive flag: backend can deactivate a virtual card without touching the phone
- Dual auth on nfc_pay: JWT (cashier identity) + X-Device-Token (device identity) — both required
- Physical RFID and NFC virtual card share the same Money Account (MoneyCardNumber link)

**H2: Troubleshooting**
- "NFC registration fails 401" → student must be logged in first (session token required)
- "NFC pay fails: virtual card not found" → check VirtualCards sheet, IsActive='TRUE', MoneyCardNumber linked correctly
- "APDU not recognized by terminal" → verify AID in apdu_service.xml matches F049494F4E41 exactly
- "Balance not deducted" → check X-Device-Token header matches DeviceToken in VirtualCards sheet
