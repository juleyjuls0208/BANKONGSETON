### R004 — SMS Notifications (Twilio)
- Class: integration
- Status: active
- Description: Parents receive SMS for purchases, loads, and low-balance events if TWILIO_* env vars are configured
- Why it matters: Philippine parents often have unreliable internet; SMS is more reliable than email

- Primary owning slice: M001/S02
- Supporting slices: none
- Validation: unmapped

### R005 — Cashier Account Management

- Why it matters: Hardcoded cashier/cashier123 is a security risk; multi-station deployment needs named accounts
- Source: user
- Primary owning slice: M001/S03
- Supporting slices: none
- Validation: unmapped
- Notes: cashier_routes.py line 62 has hardcoded credentials

### R006 — Transaction Void / Correction
- Class: operability
- Status: active

- Source: user
- Primary owning slice: M001/S03
- Supporting slices: none
- Validation: unmapped
- Notes: No void endpoint exists anywhere; needs new API + admin UI

### R007 — Offline Cashier Queue
- Class: continuity
- Status: active
- Description: Cashier can process sales when Google Sheets is unreachable; transactions buffer locally in SQLite and sync when connectivity is restored
- Why it matters: A Google Sheets outage currently hard-blocks all sales
- Source: user
- Primary owning slice: M001/S04
- Supporting slices: none
- Validation: unmapped
- Notes: resilience.py has an in-memory WriteQueue that loses data on restart; needs SQLite persistence

### R008 — Cashier Quick-Pay Shortcut
- Class: primary-user-loop
- Status: active
- Description: Cashier can tap a product's "Quick Pay" button to bypass cart — scan card immediately for single-item purchases
- Why it matters: Most lunch-rush transactions are one item; forcing cart flow for every purchase slows the queue
- Source: user
- Primary owning slice: M001/S04
- Supporting slices: none
- Validation: unmapped
- Notes: cashier_index.html has cart flow only; no single-item shortcut

### R009 — Push Notification for Every Transaction
- Class: primary-user-loop
- Status: active
- Description: Student/parent receives FCM push notification for every purchase and every load, not just low-balance events
- Why it matters: Parents want real-time visibility into every spend, not just alerts
- Source: user
- Primary owning slice: M001/S05
- Supporting slices: none
- Validation: unmapped
- Notes: send_purchase_push and send_load_push exist in fcm_sender.py but are not called from cashier or load_balance flows

### R010 — Android Transaction Filter
- Class: operability
- Status: active
- Description: Student can filter their transaction history by type (purchases / loads) and date range in the Android app
- Why it matters: Flat list of 100+ transactions is hard to navigate; students need to find specific purchases
- Source: user
- Primary owning slice: M001/S05
- Supporting slices: none
- Validation: unmapped
- Notes: TransactionsActivity shows a flat list with no filter controls

### R011 — Monthly Budget Auto-Reset
- Class: primary-user-loop
- Status: active
- Description: Budget resets automatically on the 1st of each month; if no budget is set for the new month, app prompts student to set one
- Why it matters: Budget limit persists across months making it useless as a monthly allowance tracker
- Source: user
- Primary owning slice: M001/S05
- Supporting slices: none
- Validation: unmapped
- Notes: SecureStorage stores budget_limit as a float with no month key; calcMonthlySpend already filters by current month

### R012 — Lost Card Status Feedback (Android)
- Class: primary-user-loop
- Status: active
- Description: When student reports a lost card in-app, they see a pending status; when admin replaces the card, student receives FCM notification and in-app status updates to resolved
- Why it matters: Student is currently left with no feedback after reporting — they don't know if admin processed it
- Source: user
- Primary owning slice: M001/S05
- Supporting slices: none
- Validation: unmapped
- Notes: bannerLostCard in HomeActivity shows the banner but has no status tracking post-report

### R013 — Daily Low-Balance Batch Email
- Class: operability
- Status: active
- Description: Every day at a configurable time, the system emails parents of all students with balance below the LOW_BALANCE_THRESHOLD
- Why it matters: Individual low-balance alerts fire per-transaction; a daily digest catches students who haven't transacted recently but have low balance
- Source: user
- Primary owning slice: M001/S06
- Supporting slices: none
- Validation: unmapped
- Notes: low_balance_students() endpoint exists in admin_dashboard.py; needs APScheduler or threading.Timer scheduler
